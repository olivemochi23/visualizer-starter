# Visualizer Starter

HTMLビジュアライゼーションの出力テンプレート。

常に日本語で応答する。

## ビジュアル化

デザインは全て `frontend-design` スキルに委任する。
カスタムデザイン制約は設けない。
スライド形式の可視化を作るときは `visualizer-slide-site` スキルを使い、右端プレビュー付きの共通スライド機構を維持する。
スライド形式では **`.slide` 自体を 16:9 の画面全体スライドとして扱う**。内側に `.stage` / `.slide-inner` のような 16:9 コンテナや固定キャンバスを作らない。
スライド内の情報は画面全体に対してレイアウトし、中央カードの中に押し込めない。PCブラウザ前提で、見切れや右下への食い込みを避ける。

## DESIGN.md 参照

デザインの方向性が指定された場合や、既存ブランドの視覚言語を参考にしたい場合は、必要に応じて `vendor/awesome-design-md/design-md/<brand>/DESIGN.md` を読む。
`DESIGN.md` は参考資料であり、商標やブランドそのものを複製するためではなく、配色・余白・タイポグラフィ・密度・UIの作法を理解するために使う。

## 出力ルール

- ビジュアル化ごとに **カテゴリフォルダ配下** に kebab-case のフォルダを作成
- カテゴリ: `reviews/` `analysis/` `personal/` `guides/` `tech/` `misc/`
- 各フォルダに `index.html` を配置（1フォルダ1ファイルが基本）
- 既存フォルダの内容は変更しない
- HTMLファイルを作成・編集した後は `npx prettier --write <ファイルパス>` を実行してフォーマットすること
- 新規作成後は `npm run generate:index` を実行してトップページを更新すること

## カテゴリ判定

1. 製品のレビュー・比較 → `reviews/`
2. リスク分析・コスト計算・契約・ビジネス判断 → `analysis/`
3. 人間関係・性格分析・個人ダッシュボード → `personal/`
4. ハウツー・スキルガイド・チュートリアル → `guides/`
5. 技術図解・ツール比較・アーキテクチャ → `tech/`
6. 上記に当てはまらない → `misc/`

ユーザーがカテゴリを指定した場合はそちらを優先する。
